## Контрольні питання:
### 1. Основи MLOps та його компоненти

**MLOps (Machine Learning Operations)** — це методологія та набір практик, що адаптують принципи DevOps для специфічних потреб проєктів машинного навчання. Вона виникла як перетин трьох дисциплін: машинного навчання, розробки програмного забезпечення та операційної діяльності.  

#### Проблеми, які вирішує MLOps:
* **Відсутність відтворюваності:** складність отримання однакових результатів при повторних запусках через зміни в даних чи оточенні.
* **Технічний борг:** накопичення складних залежностей між кодом, даними та конфігураціями.
* **Дрейф даних (Data Drift):** погіршення точності моделі з часом через зміну характеристик вхідних даних у реальному світі.
* **Складність розгортання:** великий розрив між розробкою моделі в ноутбуці та її запуском як надійного сервісу.
* **Відсутність моніторингу:** складність відстеження якості роботи моделі після її виходу в продакшен.


#### П’ять основних компонентів MLOps-інфраструктури:
Відповідно до типової архітектури MLOps-пайплайнів, виділяють такі ключові елементи
1. **Пайплайни даних (Data Pipelines):** забезпечує автоматизований збір, валідацію, очищення та трансформацію даних для навчання.
2. **Feature Store:** централізоване сховище для зберігання, версіонування та спільного використання ознак (features) між різними моделями.
3. **Model Registry:** каталог для зберігання натренованих моделей, їх версій та супутніх метаданих.
4. **Інфраструктура розгортання (Serving Infrastructure):** інструменти для виводу моделі в продакшен (наприклад, як API через Docker/Kubernetes).
5. **Система моніторингу та логування:** інструменти для відстеження продуктивності моделі, дрейфу даних та технічного стану системи.
   
### 2. Чому Git не підходить для версіонування великих датасетів? Які компоненти ML-проекту потребують версіонування?
#### Чому Git не підходить для великих датасетів?
Хоча Git є стандартом для версіонування коду, він має суттєві обмеження при роботі з даними в машинному навчанні:
* **Обсяг даних (Size):** ML-датасети можуть займати гігабайти або терабайти, що робить їх пряме зберігання в Git-репозиторії неефективним — операції `clone`, `push` та `pull` стають занадто повільними.
* **Робота з бінарними файлами:** дані часто представлені бінарними форматами (зображення, аудіо, відео), для яких Git не може проводити ефективний аналіз відмінностей (diff) між версіями.
* **Частота змін:** у активних проєктах датасети оновлюються дуже часто (корекція анотацій, нові приклади), що призводить до вибухового росту розміру .git директорії через зберігання історії всіх змін.
* **Обмеження платформ:** популярні хостинги (GitHub, GitLab) мають жорсткі ліміти на розмір одного файлу.

#### Які компоненти ML-проєкту потребують версіонування?
Для забезпечення повної відтворюваності результатів необхідно версіонувати:
1. **Код (Code):** скрипти для підготовки даних, архітектура моделі, тренувальні цикли.
2. **Дані (Data):** як сирі (raw) вхідні дані, так і оброблені (processed) набори, на яких безпосередньо навчалася модель.
3. **Моделі (Models):** отримані в результаті навчання ваги, серіалізовані об'єкти та супутні артефакти.
4. **Конфігурації (Hyperparameters):** всі налаштування експерименту (learning rate, кількість епох, архітектурні параметри).
5. **Середовище виконання (Environment):** версії бібліотек (requirements.txt, conda env), Docker-образи та характеристики обчислювальних ресурсів.
6. **Схеми даних (Data Schemas):** описи структури та типів даних для контролю їх цілісності.

### 3. Що зберігається в .dvc файлах і навіщо? Поясніть різницю між dvc add і git add.

**.dvc файл** — це легкий текстовий файл у форматі YAML, який створюється для кожного великого об'єкта (файлу або директорії), що відстежується через DVC.
**В середині .dvc файла міститься:**
* **outs (outputs):** список відстежуваних файлів.
* **md5:** унікальний хеш (контрольна сума) вмісту файлу, що слугує його ідентифікатором версії.
* **size:** розмір файлу в байтах.
* **path:** шлях до файлу в робочій директорії.

Замість того, щоб зберігати важкі дані в Git, ми зберігаємо там лише ці маленькі .dvc файли. Вони працюють як **посилання (покажчики)**: Git відстежує історію змін текстового .dvc файлу, а DVC за записаним у ньому хешем точно знає, яку саме версію реальних даних потрібно завантажити зі сховища.

#### Різниця між `dvc add` і `git add`

Команда **`dvc add`** працює з великими бінарними файлами (датасети, моделі). Вона обчислює контрольну суму (MD5-хеш) файлу, переміщує його оригінальний зміст у локальний кеш DVC (`.dvc/cache`), а в робочій директорії створює легкий текстовий файл-покажчик з розширенням `.dvc`. Одночасно з цим оригінальний великий файл автоматично додається до `.gitignore`, щоб він ніколи не потрапив у Git-репозиторій.

Команда **`git add`** використовується для фіксації в репозиторії лише створеного `.dvc` файлу та оновленого `.gitignore`. Оскільки файл метаданих дуже малий, Git легко відстежує його зміни. Таким чином, замість зберігання гігабайтів даних, Git зберігає лише посилання на конкретну версію цих даних, що дозволяє синхронізувати версії коду з відповідними версіями датасетів без перевантаження репозиторію.  

**Типовий ланцюжок команд:**
1. `dvc add data/raw.csv` (DVC бере файл під контроль).
2. `git add data/raw.csv.dvc .gitignore` (Git починає відстежувати посилання на цей файл).
3. `git commit -m "Add raw dataset version 1"` (Фіксація версії).

### 4. Компоненти та термінологія MLflow

**MLflow** — це платформа з відкритим вихідним кодом для управління повним життєвим циклом машинного навчання.

#### Чотири основні компоненти MLflow:
1. **MLflow Tracking:** API та UI для логування параметрів, версій коду, метрик та артефактів під час роботи моделі.
2. **MLflow Projects:** стандартний формат для пакування коду ML-проєктів з метою забезпечення відтворюваності на різних платформах.
3. **MLflow Models:** формат для пакування моделей, який дозволяє використовувати їх у різних інструментах (наприклад, Apache Spark або REST API).
4. **MLflow Model Registry:** централізоване сховище моделей для спільного управління версіями, переходами станів (Stage) та анотаціями.

#### Основні терміни:
* **Run (Запуск):** одиничний цикл виконання коду моделі. Кожен запуск фіксує конкретний набір параметрів, метрик та артефактів.
* **Experiment (Експеримент):** основна одиниця організації, що групує декілька запусків (Runs) за певною темою або завданням.
* **Parameter (Параметр):** вхідні дані для налаштування моделі, що мають формат «ключ-значення» (наприклад, `learning_rate=0.01` або `num_estimators=100`). Параметри є константами протягом одного запуску.
* **Metric (Метрика):** числовий показник результативності моделі (наприклад, `accuracy`, `RMSE`, `F1-score`), який може змінюватися в процесі навчання або фіксуватися в кінці.
* **Artifact (Артефакт):** вихідні файли будь-якого формату, що генеруються під час запуску (ваги моделі у форматі `.pkl` чи `.h5`, графіки навчання у `.png`, звіти у `.csv`).

### 5.  Що таке scikit-learn Pipeline? Чому Pipeline запобігає data leakage?

**scikit-learn Pipeline** — це інструмент, що дозволяє об'єднати послідовність кроків обробки даних та модель машинного навчання в єдиний об'єкт. Він автоматично передає вихідні дані одного етапу на вхід наступному, дотримуючись суворого порядку дій.

#### Чому Pipeline запобігає data leakage (витоку даних)?

Data leakage виникає, коли інформація з тестового набору даних неявно використовується при навчанні моделі, що призводить до отримання надто оптимістичних, але невірних результатів.

Pipeline запобігає цьому завдяки таким механізмам:

1. **Ізоляція етапів fit та transform:** При використанні крос-валідації або розбитті даних, Pipeline гарантує, що метод `.fit()` (наприклад, для обчислення середнього значення при масштабуванні) викликається **лише на тренувальних фолдах**. 

2. **Автоматичне застосування параметрів:** Параметри, отримані під час навчання, зберігаються в середині Pipeline і застосовуються до тестових даних через `.transform()`. Таким чином, статистика тестової вибірки залишається невідомою для моделі до моменту самого тестування.

3. **Цілісність процесу:** Оскільки всі кроки (очищення, створення ознак, масштабування) зафіксовані в одному об'єкті, неможливо випадково обробити тренувальні та тестові дані за різними правилами.
 
 ### 6. Обґрунтуйте структуру директорій вашого проекту. Чому важливо розділяти raw/processed/external дані?

Чітка структура проекту забезпечує відтворюваність експериментів та полегшує спільну роботу над кодом. Типова структура MLOps-проєкту виглядає так:
- `data/` — зберігання всіх версій даних.
- `models/` — збережені ваги та серіалізовані моделі.
- `src/` — вихідний код (скрипти для обробки, навчання, інференсу).
- `notebooks/` — Jupyter ноутбуки для досліджень (EDA).

#### Чому важливо розділяти дані на Raw, Processed та External?

Розділення даних за категоріями є основою **Data Lineage** та відтворюваності експериментів.

1. **data/raw (Сирі дані)**
   -  Це оригінальні дані, які ніколи не підлягають ручному редагуванню. Якщо під час обробки виникне помилка або знадобиться змінити методику підготовки ознак, то завжди можна перезапустити процес з початкової точки. Це гарантує, що не відбудеться втрата вхідної інформації.

2. **data/processed (Оброблені дані)** 
   -  Тут зберігаються дані після очищення, заповнення пропусків та генерації ознак. Їх виокремлення дозволяє не витрачати ресурси та час на повторну трансформацію при кожному запуску навчання. Це також дозволяє порівнювати різні версії обробки для однієї і тієї ж сирої бази.

3. **data/external (Зовнішні дані)**
   -  Сюди потрапляють сторонні датасети (наприклад, довідники адрес, курси валют або погода), які не є частиною  основного збору даних. Це відокремлює залежності від сторонніх сервісів від власних даних, що полегшує управління ліцензіями та оновленнями.

Оскільки великі дані та моделі не зберігаються безпосередньо в Git, процес отримання повної робочої копії проєкту складається з двох кроків:

### 8. Колега хоче відтворити ваш експеримент. Які інструкції дати? Потрібно повернутись до моделі тижневої давності. Як?
Щоб повністю відтворити робоче середовище та результати, колега має виконати такі кроки:
1. **Клонувати репозиторій:** `git clone <url>` та перейти в папку проєкту.
2. **Налаштувати оточення:** Встановити залежності (наприклад, `pip install -r requirements.txt`), щоб версії бібліотек збігалися.
3. **Отримати дані та моделі:** Виконати `dvc pull` для завантаження всіх артефактів, що відповідають поточному стану коду.
4. **Запустити відтворення:** Використати команду `dvc repro` (якщо налаштовані `dvc.yaml` пайплайни) або запустити відповідний скрипт тренування.

#### Як повернутися до моделі тижневої давності?
Повернення до старого стану системи — це двоетапний процес, оскільки потрібно синхронізувати і код, і дані.

1. **Знайти та активувати потрібний стан у Git:**
   Знайдіть хеш коміту, що був зроблений тиждень тому (через `git log`), та перейдіть на нього:
   ```bash
   git checkout <commit_hash_of_last_week>
   ```
2.**Синхронізувати дані та моделі через DVC:**
Після зміни стану Git потрібно оновити фізичні файли в робочій директорії, щоб вони відповідали новим  посиланням:
  ```bash
  dvc checkout
  ```
Ця команда замінить поточні файли в data/ та models/ на ті версії, які були актуальні на момент вибраного коміту.

### 9. Порівняйте ваші результати з та без Pipeline. Який вплив має кожен гіперпараметр на метрики?

**Без Pipeline:** Спочатку препроцесинг застосовувався до всього датасету до його розбиття на train/test. Це призводило до **витоку даних (data leakage)**, оскільки словник формувався з урахуванням тестової вибірки. Як наслідок, метрики були необ'єктивно завищені, а модель у продакшені працювала б гірше. Крім того, доводилося зберігати векторайзер та модель як два окремі файли.

**З Pipeline:** Використання `scikit-learn Pipeline` об'єднало векторизацію та класифікатор. Тепер під час навчання  метод `fit` застосовується **виключно до тренувальної вибірки**. Отримані метрики стали чесними. Також це спростило розгортання — вся логіка обробки збережена в єдиному артефакті `pipeline.pkl`.

#### Вплив гіперпараметрів на метрики
В ході експериментів було досліджувано вплив наступних гіперпараметрів на баланс між недонавчанням та перенавчанням:

1. **`alpha` (у Multinomial Naive Bayes):**
   - Відповідає за згладжування Лапласа .
   - **Результат:** Зменшення `alpha` з дефолтного 1.0 до 0.1 (слабше згладжування) дозволило алгоритму сильніше спиратися на реальні частоти слів у текстах, що підвищило Accuracy з 0.7347 до 0.7472.

2. **`C` (у Logistic Regression та Linear SVC):**
   - Це обернений параметр сили регуляризації (чим менше `C`, тим сильніший штраф за складність).
   - **Вплив на Logistic Regression:** Збільшення `C` до 10.0 (ослаблення регуляризації) дозволило моделі точніше підлаштуватися під дані, трохи покращивши точність (до 0.7415).
   - **Вплив на Linear SVC (Найкраща модель):** Зменшення `C` до 0.5 (посилення регуляризації) дало глобально найкращий результат (**Accuracy: 0.7488**). Сильніший штраф допоміг SVM проігнорувати шумовий вплив рідкісних слів  і побудувати оптимальну гіперплощину між 20 класами.
